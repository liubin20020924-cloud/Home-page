## 什么是 CI/CD

### CI（Continuous Integration）持续集成

CI 是一种开发实践，要求开发人员频繁地（一天多次）将代码集成到共享仓库中。

**主要目标：**
- 尽早发现集成错误
- 减少集成问题
- 提高代码质量
- 自动化测试和验证

**本项目的 CI 实现：**
- 自动化单元测试
- 代码质量检查（Lint）
- 安全漏洞扫描
- 自动代码格式化

### CD（Continuous Deployment）持续部署

CD 是在 CI 通过后，自动将代码部署到生产环境的过程。

**主要目标：**
- 减少手动部署错误
- 加快发布周期
- 提高部署可靠性
- 实现零停机部署

**本项目的 CD 实现：**
- 自动化备份机制
- 智能代码拉取
- 自动依赖更新
- 服务无缝重启

## 系统架构

### 整体架构图

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│  本地开发    │────→│   GitHub     │────→│  云主机     │────→│  生产环境   │
│             │     │              │     │             │     │             │
│ - 开发代码   │     │ - 代码仓库  │     │ - 部署目标 │     │ - 运行应用   │
│ - 本地测试   │     │ - CI/CD     │     │ - 代码拉取   │     │ - 服务访问   │
│ - 提交代码   │     │ - PR 审核    │     │ - 自动部署   │     │             │
│             │     │              │     │             │     │             │
└─────────────┘     └──────────────┘     └─────────────┘     └─────────────┘
                            ↓
                    Webhook/SSH 通知
```

### 部署方式

**主要方式：Webhook 触发**
- GitHub Actions 推送事件触发
- 通过 HTTP POST 通知云主机
- 轻量级，响应快速
- 依赖网络连接

**备用方式：SSH 直接执行**
- GitHub Actions 通过 SSH 连接云主机
- 直接执行部署脚本
- 可靠性高，不依赖外部网络
- 适合网络不稳定环境

## 核心组件

### 1. GitHub Actions 工作流

**文件位置**: `.github/workflows/ci-cd.yml`

**主要任务：**

| 任务 | 说明 | 耗时 |
|------|------|------|
| 代码检查 | 检查代码质量、语法 | ~1分钟 |
| 安全检查 | 扫描安全漏洞 | ~2分钟 |
| 单元测试 | 运行所有测试用例 | ~3分钟 |
| 自动合并 | PR 检查通过后自动合并 | ~1分钟 |
| 部署通知 | 通知云主机开始部署 | ~30秒 |

### 2. 云主机 Webhook 接收器

**文件位置**: `scripts/webhook_receiver_github.py`

**功能：**
- 接收 GitHub Webhook 通知
- 验证请求签名
- 触发部署脚本
- 记录部署日志
- 提供版本查询 API

**API 端点：**
- `POST /webhook/github` - 接收部署通知
- `GET /webhook/version` - 查询当前版本
- `GET /webhook/logs` - 查询部署日志
- `GET /webhook/health` - 健康检查

### 3. 自动部署脚本

**文件位置**: `scripts/deploy.sh`

**执行步骤：**

1. **创建备份**
   - 保存当前代码版本
   - 保留最近5个备份
   - 自动清理过期备份

2. **拉取代码**
   - 使用智能拉取脚本
   - 测试 GitHub 和 Gitee 连接速度
   - 选择最快的源拉取

3. **更新依赖**
   - 安装新的 Python 包
   - 更新 requirements.txt
   - 处理依赖冲突

4. **重启服务**
   - 优雅停止应用
   - 更新代码文件
   - 启动应用服务
   - 健康检查验证

### 4. 智能拉取脚本

**文件位置**: `scripts/smart-pull.sh`

**智能选择逻辑：**
```bash
# 测试 GitHub 连接速度
SPEED_GITHUB = measure_speed(github.com)

# 测试 Gitee 连接速度
SPEED_GITEE = measure_speed(gitee.com)

# 选择最快的源
if SPEED_GITHUB > SPEED_GITEE:
    pull_from = "Gitee"
else:
    pull_from = "GitHub"
```

### 5. 回滚脚本

**文件位置**: `scripts/rollback.sh`

**功能：**
- 列出所有可用备份
- 选择指定版本回滚
- 自动停止服务
- 恢复备份文件
- 重启服务

## 工作流程

### 完整部署流程

```
1. 本地开发
   ↓
2. git push origin feat/xxx
   ↓
3. 创建 Pull Request (feat/xxx → main)
   ↓
4. GitHub Actions 自动触发
   ├─ 运行测试
   ├─ 代码检查
   └─ 安全扫描
   ↓
5. 检查通过，自动合并到 main
   ↓
6. GitHub Actions 触发部署
   ├─ Webhook 通知云主机 (主要)
   └─ SSH 执行部署 (备用)
   ↓
7. 云主机执行 deploy.sh
   ├─ 创建备份
   ├─ 拉取代码
   ├─ 更新依赖
   └─ 重启服务
   ↓
8. 部署完成，应用更新
```

### 分支策略

| 分支 | 用途 | CI/CD | 自动部署 |
|------|------|--------|-----------|
| `main` | 生产环境 | ✅ | ✅ |
| `develop` | 开发环境 | ✅ | ❌ |
| `feat/*` | 功能开发 | ✅ | ❌ |
| `fix/*` | Bug 修复 | ✅ | ❌ |
| `hotfix/*` | 紧急修复 | ✅ | ❌ |

**部署触发条件：**
- 代码合并到 `main` 分支
- 且所有 CI 检查通过
- 且 PR 标签包含 `auto-merge`

## 部署策略

### 备份策略

**保留策略：**
- 保留最近5个版本备份
- 每次部署前自动备份
- 备份命名：`Home-page_YYYYMMDD_HHMMSS`
- 备份位置：`/var/backups/Home-page/`

**自动清理：**
- 删除超过7天的备份
- 保留磁盘空间
- 避免备份累积

### 版本回滚

**回滚场景：**
- 新版本严重 Bug
- 部署失败
- 性能严重下降
- 需要快速恢复

**回滚步骤：**
1. 停止当前服务
2. 选择备份版本
3. 恢复备份文件
4. 重启服务
5. 验证功能正常

### 监控和告警

**监控项：**
- 部署成功率
- 部署耗时
- 服务启动时间
- 应用健康状态

**告警机制：**
- 部署失败邮件通知
- 服务异常告警
- 响应时间超时告警

## 版本管理

### 版本号规则

采用语义化版本号（Semantic Versioning）：

```
主版本号.次版本号.修订号

示例：v2.8.0
     │  │  └─ 修订号（Bug修复）
     │  └─ 次版本号（新功能）
     └─ 主版本号（重大变更）
```

**更新规则：**
- 主版本号：不兼容的 API 变更
- 次版本号：向下兼容的功能性新增
- 修订号：向下兼容的问题修正

### 变更日志

**文件位置**: `CHANGELOG.md`

**记录格式：**

## 安全机制

### 1. 签名验证

**Webhook 签名：**
- 使用 HMAC-SHA256 算法
- GitHub 自动计算签名
- 云主机验证签名
- 防止伪造请求

### 2. 访问控制

**GitHub Secrets：**
- 敏感信息存储在 GitHub Secrets
- 不暴露在代码中
- 定期轮换密钥
- 最小权限原则

**SSH 密钥管理：**
- 使用 ed25519 算法（更安全）
- 私钥权限 600
- 公钥权限 644
- 定期更换密钥

### 3. 网络安全

**代理配置：**
- 国内环境配置代理
- 仅 GitHub 使用代理
- 其他访问直连
- 避免代理泄露

**防火墙规则：**
- 仅开放必要端口
- Webhook 端口限制源 IP
- SSH 端口使用密钥认证
- 定期审计访问日志

## 最佳实践

### 开发建议

1. **小步快跑**
   - 频繁提交，每次提交功能单一
   - 便于代码审查
   - 减少冲突概率

2. **提交前测试**
   - 本地运行所有测试
   - 确保功能正常
   - 避免 CI 失败

3. **遵循提交规范**
   - 使用 Conventional Commits 格式
   - 清晰描述变更内容
   - 关联相关 Issue

### 部署建议

1. **分阶段部署**
   - 先部署到测试环境
   - 验证功能正常
   - 再部署到生产环境

2. **灰度发布**
   - 新功能先小范围测试
   - 收集用户反馈
   - 逐步扩大范围

3. **准备回滚**
   - 每次部署前确保可以快速回滚
   - 准备回滚预案
   - 减少故障影响
